    Returns both ScanResult (for backward compat with existing resolve flow) AND Vec<DiscoveredRef> for the new typed extraction pipeline.

    Implementation strategy using pulldown-cmark events:

    1. **Setup**: Create pulldown-cmark Parser with Options:
       ```rust
       use pulldown_cmark::{Parser, Options, Event, Tag, TagEnd, CowStr};
       let mut opts = Options::empty();
       opts.insert(Options::ENABLE_HEADING_ATTRIBUTES);
       opts.insert(Options::ENABLE_WIKILINKS);
       let parser = Parser::new_ext(body, opts);
       ```
       NOTE: Do NOT enable ENABLE_YAML_STYLE_METADATA_BLOCKS (conflicts with split_frontmatter per locked decision).

    2. **State tracking**: Track nesting depth to know when we're inside code blocks:
       ```rust
       let mut in_code_block = false;
       let mut in_inline_code = false;
       let mut current_heading: Option<String> = None;
       let mut text_accumulator = String::new();
       let mut block_start_offset: usize = 0;
       ```

    3. **Event loop** using `parser.into_offset_iter()` for byte offsets:
       - `Event::Start(Tag::CodeBlock(_))` -> set in_code_block = true
       - `Event::End(TagEnd::CodeBlock)` -> set in_code_block = false
       - `Event::Code(_)` -> skip (inline code, single event, no Start/End)
       - `Event::Start(Tag::Heading { level, .. })` -> start collecting heading text
       - `Event::End(TagEnd::Heading(_))` -> finalize heading: create Section handle in graph
       - `Event::Start(Tag::Link { dest_url, .. })` or wiki-link handling:
         - For standard links: dest_url is the target. If it ends with .md or looks like a label, create DiscoveredRef
         - For wiki-links: pulldown-cmark 0.13 uses `Tag::Link` with `LinkType::WikiLink`
       - `Event::Text(text)` (when NOT in code block):
         - If collecting heading: append to current_heading text
         - Otherwise: accumulate text for the current block element
       - `Event::Html(html)` and `Event::InlineHtml(html)`:
         - Scan the HTML content with the existing regex patterns (LABEL_RE, SECTION_REF_RE, FILE_PATH_RE)
         - This handles EXTRACT-11 (HTML block scanning)
       - `Event::Start(Tag::Paragraph)` / other block starts: reset text_accumulator, record block_start_offset
       - `Event::End(TagEnd::Paragraph)` / other block ends: run regex patterns on accumulated text (EXTRACT-08: concatenate text events per block)

    4. **Text scanning** (called when block ends, on accumulated text):
       Apply the same regex patterns as the old scanner (LABEL_RE, SECTION_REF_RE, FILE_PATH_RE, version patterns) to the concatenated text. Use infer_edge_kind_from_line for edge kind inference. For each match:
       - Create LabelCandidate (for ScanResult backward compat)
       - Create DiscoveredRef with RefSource::Body and SourceSpan using line_index.offset_to_line(block_start_offset)

    5. **Link extraction** (EXTRACT-09):
       When processing Link events:
       - Standard `[text](target.md)` links: classify dest_url with the same logic as frontmatter (label pattern, .md extension, section ref pattern)
       - Wiki-links `[[target]]`: classify the target text
       - Create DiscoveredRef with RefSource::Body, appropriate RefHint, and SourceSpan
       - Also add to ScanResult.file_refs or label_candidates for backward compat

    6. **File path rejection rules**: Apply the same rejection rules as old scanner:
       - Reject URL fragments (preceded by "://")
       - Reject version-dot fragments (preceded by '.')
       - Reject hyphen-prefixed fragments
       - Reject mid-word matches (preceded by alphanumeric)
