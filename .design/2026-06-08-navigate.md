# Navigation: lineage + current-neighborhood — 2026-06-08

**For review.** The opening `navigate(root, intent)` design was put through
currency's loop — *simulated on a real corpus first* — and the simulation broke
it. This is the revised, simulation-grounded design: two cheap, real first cuts
(**lineage** + **currency-ranked neighbors**), with the grand topical path-finder
deferred to a substrate that doesn't exist yet. Going to codex for adversarial
review before any implementation.

## What the simulation found (murail + anneal, 2026-06-08)

The opening design assumed navigate would *walk author-declared typed edges toward
intent*. The real edge graph won't feed that:

- **~95% of edges are `Cites`** (murail 5857/6000; anneal 1119/1188), plus a
  sparse directional skeleton (`Affects` 103, `DependsOn` 27, `Supersedes` 13) and
  **no `Refines`/parent at all**.
- **Index hubs dominate** — `LABELS.md` alone emits **1822** citation edges over
  464 files. Transitive `Cites`-walking routes almost any path through a hub in
  ~2 hops (`X → LABELS → Y`): **non-trails**.
- **`context` already emits the edge-neighborhood** — 213 neighbor rows for a
  3-hit perf query, now *with* currency dispositions. Navigate-as-neighborhood is
  largely already shipped.
- **The one richly-navigable structure is lineage** — the `Supersedes`/version
  chains are deep, ordered, and clean (`v6→v8→v9→v11→…→v17`; `cell-graph` +
  `05-30 → 05-31`).

So `navigate(root, intent)` as edge-walking would produce hub-routed garbage *and*
duplicate `context`. The simulate-first loop caught it before we built it.

## The general walk already exists — in the language

A generic "walk edges of kind K transitively from root" is a parameterized
recursive Datalog rule, and anneal already runs them (we used one in the currency
sim):

```
? where reach(h, x) := *edge{from: h, to: x, kind: "Supersedes"}.
  where reach(h, x) := reach(h, mid), *edge{from: mid, to: x, kind: "Supersedes"}.
  reach("…", x).
```

So the *general primitive* is already on the surface for power users via
`anneal -e`. There is **no missing generalized capability** — the recursion
engine is it. The open question is only which walks earn a *curated verb*.

## The decision: curated verbs per meaningful walk, not a generic `navigate(kind)`

A kind-parameterized `navigate(root, kind)` verb would mostly expose footguns
(`walk(Cites)` = hub blowup). Instead, a walk earns a curated verb only when it is
**both structurally sound and semantically distinct** — and then the verb carries
*that kind's* semantics. anneal already follows this pattern:

| edge kind | structure | curated verb | semantics it carries |
|---|---|---|---|
| `DependsOn` | DAG | **`handle --impact`** (exists) | blast radius |
| `Supersedes` | chain → head | **`lineage`** (this) | currency: current head / superseded / route-to-head |
| `Cites` | dense + hubs | **currency-ranked `context` neighbors** (this) | current neighborhood, hubs penalised |
| topical (cross-cut) | — | *deferred* | needs the clustering substrate |

`lineage` is not `walk(Supersedes)` — it is `walk(Supersedes)` **+ currency
semantics**: it knows the chain has a *current head*, flags each node
current/superseded, and routes a stale node to the live head. That UX is exactly
what a generic walk can't give. (The trust invariant applies: don't expose a
"navigation" where the oracle/structure is garbage — expose the walks that earn it.)

## First cuts

### (a) `lineage` — the Supersedes verb (currency's chain view)
For a handle, return its supersession lineage with currency dispositions and the
current head:
```
$ anneal lineage <handle>
```
- Superseded handle → flag `⊘ superseded`, show the chain, name the **current
  head**, and point to it (`anneal read <head>`).
- Current handle → show its lineage tail (what it displaced) and confirm
  `current_head`.
- Built on the `currency_*` predicates already shipped + the recursion engine —
  no new extraction. Sibling of `--impact`.

### (b) currency-ranked `context` neighbors — the Cites surface
`context`/`search` already emit the neighborhood; rank it instead of dumping it:
- order neighbors by **currency** (current_head/current first, superseded
  collapsed-but-reachable) × **relevance** × a **hub penalty** (down-rank
  `LABELS`/label-handles and other high-degree index nodes);
- group as *current* / *in-flight (draft)* / *superseded (history)* / hidden hubs.
Turns 213 raw neighbors into a small current orientation map. Uses existing facts
(`Cites` edges + currency + lifecycle + degree).

## Acceptance (grounded in real murail data)
- `lineage 2026-05-30-unified-parametric-perf` → flags superseded, shows the
  chain, names current head `2026-05-31-program-space`, points to it.
- `lineage formal-model/murail-formal-model-v14` → `v6…v14…v17`, head = `v17`.
- `context "parametric perf"` neighbor section → `2026-06-08-corner-reachability`
  (current) + `v17 §cost-model` (current) surface; `05-30`/`cell-graph`
  collapsed under superseded; `LABELS`/`WS-*` hubs hidden.
- Non-navigation surfaces byte-identical; murail `check` 0 E001.

## Deferred (off the critical path — proven by the sim)
- **Topical navigation** (`root → intent` across topics) — needs the
  **topic-clustering substrate** (communities over `Cites`), the *same* substrate
  the unmarked-currency frontier needs. Build it once, then both. Until then the
  general recursive walk in the language covers power-user cases.
- **More curated walk-verbs** — only if a future edge kind becomes both rich and
  semantically distinct (e.g. authored `Refines` chains).

## Open questions for review
- `lineage` as a new verb vs a `--lineage` mode on `read`/`handle` (surface
  economy — we already have `--impact` on `handle`).
- Hub-penalty shape for the neighbor ranking — degree threshold vs an explicit
  index-handle class (`LABELS`, `*-` labels). Pin it by simulating the ranking.
- Lineage with branches/merges (cell-graph + 05-30 both → 05-31): chain or small
  tree? cycles guard.
- Does `lineage` subsume part of `--explain`/trail, or stay distinct?
