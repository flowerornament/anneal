# ji1s EEP-48 extraction sim — 2026-06-10

This is the blocking Elixir twin of the bqqc rustdoc sim. It runs out of
band: `scripts/simulate-eep48-extraction.exs` reads compiled EEP-48 `Docs`
chunks from Herald's BEAM artifacts and projects the aggregate shape an
`anneal-code` Elixir adapter would emit. No adapter code, product predicates,
or eval-time artifact generation are involved.

Row-level module data stays local because Herald is private:
`/tmp/anneal-ji1s-herald-eep48.json`. This document commits only aggregate
evidence and the contract verdict.

Command:

```bash
scripts/simulate-eep48-extraction.exs \
  --root "$HERALD_ROOT" \
  --app herald \
  --build-env dev \
  --output /tmp/anneal-ji1s-herald-eep48.json
```

## Doc links

The EEP-48 chunk stores markdown docs, not a structured doc-link graph. This
sim found only 4 explicit markdown links and 2 unique targets in the compiled
docs. So the contract survives only if the Elixir adapter parses doc markdown
to produce `Cites`; EEP-48 does not hand us structured ExDoc link edges.

This is a contract bend, not a break: doc links are still citations, but the
adapter must own markdown-link extraction over EEP-48 doc text.

## Identity stressors

| measure | count |
| --- | ---: |
| source files with modules | 896 |
| source files with more than one module | 125 |
| modules in multi-module files | 294 / 1,065 (27.6%) |

This validates the two-key identity rule. `path#Module` works for multiple
modules in one file, but path-sensitive handle identity alone is not enough
for movement/re-export reasoning. `code.qualified_name` metadata is required
from day one for Elixir too.

## Contract verdict

The locked §2 contract survives Elixir unmodified, with three operational
clarifications:

1. **Doc-link edges require parsing EEP-48 markdown doc text.** EEP-48 is the
   artifact of record for docs/signatures/metadata, but it is not itself a
   structured edge list.
2. **Granularity policy matters more for Elixir than expected.** A large app
   emits many hidden/generated/framework member docs. The first Elixir adapter
   should start with modules plus documented/signature-bearing members, and
   keep hidden/generated classification explicit so it can downscope product
   surfaces without deleting evidence.
3. **Two-key identity is mandatory.** Multi-module files are common enough
   that `path#<adapter-local-id>` is necessary, while qualified-name metadata
   is necessary for moves and name-based joins.

No locked rule needs to change before slice 1. The adapter contract should
stay:

- handles keyed by path-prefixed opaque adapter ids;
- stable qualified names in metadata;
- `UsesType` coverage-graded;
- `Implements` kind/evidence preserving behaviours and protocol impls;
- docs + signatures only for v1 content;
- source classification as metadata, not status or visibility.

## Recommendation

Proceed with slice 1, the CR-D8 `assertion_date`/`assertion_revision` schema
change. The Elixir sim does not invalidate the bilingual contract. It does
raise one implementation priority for the later Elixir adapter: keep the
artifact layer honest about hidden/generated volume and make markdown-doc-link
parsing an explicit part of EEP-48 ingestion rather than assuming structured
links exist.
